# RT-SC · Phase 4e.6.2 — Colle session bug + period-lock

Two real fixes, both from your sharp testing.

## 1. "Session prof invalide" — bug fix

**Diagnosis.** I was reading `s.profil?.uid` from the auth store. Looking at the actual `Professeur` interface in `models.ts`:

```ts
export interface Professeur {
  id: string         // ← exists
  nom: string
  email: string
  matieres: string[]
  classesIds: string[]
  // ... no uid field
}
```

There's no `uid` on the Professeur Firestore profile — it's just `id`. The Firebase Auth UID lives on the AUTH user (`s.user?.uid`), not on the profile doc. NotesGrid was already doing this correctly:

```ts
const profilUid = useAuthStore((s) => s.user?.uid)
```

I just followed the wrong pattern in ModalGiveColle. One-line fix.

After this:
- Tap any colle button → modal opens
- Pick hours, optionally motif → tap "Enregistrer la colle"
- Toast: "Colle de Xh enregistrée pour [élève] ([période])."
- No more "Session prof invalide"

## 2. Period-lock for colles — feature, important

You asked: *"since colle affect the calcul de la moyenne générale, shouldn't they be locked too when the period is locked too?"*

You're right. Here's the reasoning and the implementation.

### The problem if we don't lock

A colle changes the conduite calculation. Conduite affects the moyenne générale on the bulletin. The bulletin is generated by the PP at a specific moment in time, capturing `noteConduite` as a baked-in value on the doc.

If a prof adds a colle AFTER the bulletin is generated:
- The Heures de colle widget on the élève's Accueil shows the new colle
- BUT the printed bulletin shows the OLD conduite (from before the colle)
- The two screens disagree
- If the family disputes "but the widget shows my child has 4h of colle, why does the bulletin show full conduite?", we have an integrity problem

### The lock rule

**A colle for (élève × période) is forbidden when a Bulletin doc exists for that (élève × période).**

That's the simplest, safest definition. It doesn't matter whether the bulletin is `estVerrouille` or just a draft — if the PP has generated it once, the conduite was captured. Changing colles silently would drift things.

### What about the per-matière "Clôturer" lock?

Subtle distinction. There are TWO kinds of "lock" in the bulletin module:

- **Lock A — per-matière closure** (`Note.estCloture = true`). When a prof closes their matière, those NOTES are locked. But colles aren't tied to one matière for conduite purposes — the PP could still generate the bulletin or not, other matières might still be open.
- **Lock B — bulletin generated** (Bulletin doc exists). The PP has finalized this period for this élève. Conduite is baked in. THIS is when colles must lock.

Lock A doesn't lock colles. Lock B does.

### How to add a late colle

The colle UI now shows a clear locked state explaining the workflow:

> Pour ajouter une colle à cette période :
> 1. Le PP doit d'abord supprimer les bulletins de la période (onglet Bulletins).
> 2. Ajoutez ensuite la colle ici.
> 3. Le PP regénère les bulletins de la période.

The "déverrouiller bulletins → modifier → regénérer" workflow already exists in the PP bulletin tab from Phase 4c. Same pattern as fixing a note error post-generation.

## Implementation

```ts
// ModalGiveColle.tsx
const { data: bulletinExists, isLoading: bulletinChecking } = useQuery({
  queryKey: ['colle-modal-bulletin-exists', classeId, eleveId, periode],
  enabled: open && !!classeId && !!eleveId && !!periode,
  staleTime: 30_000,
  queryFn: async () => {
    const snap = await getDoc(doc(db, bulletinDoc(classeId, eleveId, periode)))
    return snap.exists()
  },
})
const periodLocked = bulletinExists === true
```

Three render branches in ModalBody:
1. **`bulletinChecking`** — spinner, brief
2. **`periodLocked`** — `<LockedNotice/>` with explanation + workflow steps
3. **else** — the entry form

Footer adapts too: locked state shows only "Fermer" (no save button at all). Submit guard added: `if (periodLocked || bulletinChecking) return` — defense in depth even if the UI is bypassed.

The query is gated by `enabled: open` so it only fires when the modal is actually shown — no wasted Firestore reads. 30-second staleTime means rapid open/close cycles share results.
